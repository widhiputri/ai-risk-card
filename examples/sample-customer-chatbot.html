Purpose and Usage

Intended Uses, Users and Domains
A generative AI chatbot deployed on the FI's retail banking web and mobile platforms. Handles customer enquiries related to account balances, transaction history, product information, and general banking FAQs. Intended for use by retail customers aged 18 and above.

Not intended for: investment advice, loan decisioning, complaint resolution requiring human judgement, or any interaction involving customers identified as vulnerable.
Usage Guidance
The chatbot must be clearly identified as an AI system at the start of every session. A handoff to a human agent must be offered at any point upon customer request, and must be triggered automatically for: complaints, expressions of financial distress, requests for regulatory disclosures, and any topic outside the approved scope.

Maximum conversation turns per session: 20. Sessions exceeding this limit are automatically handed off to a human agent.

Outputs must not be presented as definitive financial advice. All product-related outputs must include a standard disclaimer directing customers to verify details on the FI's official website.

Techniques and Development

Main AI Techniques
Retrieval-Augmented Generation (RAG) architecture built on a third-party large language model (LLM) foundation model. The system retrieves relevant content from an FI-controlled knowledge base (product FAQs, terms and conditions, regulatory disclosures) before generating responses. This grounds outputs in verified FI content and reduces hallucination risk.
Architecture and Components
Three-layer architecture:
1. Input layer: user message is passed through an input filter (jailbreak and off-topic detection) before reaching the LLM.
2. Retrieval layer: a vector database stores embeddings of the FI's approved knowledge base. Relevant chunks are retrieved based on semantic similarity to the user query.
3. Generation layer: the LLM generates a response grounded in the retrieved chunks. Output is passed through an output filter (toxicity, policy compliance) before being returned to the user.

The foundation model is provided as a managed API by ChatCore AI. The FI owns and controls the knowledge base.
External AI Service Calls
ChatCore AI managed LLM API (text generation). Calls include the user message, retrieved knowledge base chunks, and a system prompt defined by the FI. No customer PII is included in API calls - messages are anonymised before transmission.

Risks

6
▶ Hallucination / Fabrication / Confabulation Robustness & Stability
Mitigation Strategy
RAG architecture grounds responses in FI-approved content, reducing reliance on the model's parametric knowledge. A consistency check is run on each response: the output is compared against the retrieved source chunks using BERTScore. Responses with low source similarity trigger a fallback response and human handoff.
Appropriate Use Guidance
Do not deploy the chatbot for topics not covered by the knowledge base without first adding verified content. Monitor the human handoff rate weekly - a spike may indicate the model is encountering out-of-scope topics.
▶ Prompt injection Cyber & Data Security
Mitigation Strategy
Input filtering layer screens all user messages for prompt injection patterns using a combination of regex rules and a secondary classifier model. Detected injection attempts are blocked and logged. The system prompt is hardened and not exposed to users. Knowledge base retrieval uses parameterised queries to prevent injection via retrieved content.
Appropriate Use Guidance
Review the injection attempt log monthly. Any novel injection pattern not caught by the filter must be added to the rule set within 5 business days of detection.
▶ Anthropomorphism Transparency
Mitigation Strategy
The system prompt explicitly instructs the model not to claim to be human, not to use first-person emotional language, and to identify itself as an AI when directly asked. The UI displays a persistent 'AI Assistant' label throughout the conversation. Session opening message includes a mandatory AI disclosure statement.
Appropriate Use Guidance
Any UI or system prompt change that could remove or obscure the AI disclosure must go through the AI-specific review process before deployment.
▶ Unclear output accuracy Transparency
Mitigation Strategy
All product-specific outputs (interest rates, fees, eligibility criteria) are retrieved from the knowledge base rather than generated from model weights. Retrieved content is time-stamped and the knowledge base is refreshed weekly. Outputs that cannot be grounded in retrieved content trigger a fallback response.
Appropriate Use Guidance
Ensure the knowledge base refresh process is monitored. Stale content (last updated more than 14 days ago) should trigger an alert to the content owner.
▶ Inadequate feedback and recourse mechanisms Accountability & Governance
Mitigation Strategy
Each chatbot response includes a thumbs up/down feedback button. Negative feedback is reviewed by the customer experience team weekly. Customers can escalate to a human agent at any time. A formal complaints process is accessible via a persistent link in the chat UI.
Appropriate Use Guidance
Negative feedback rate above 15% on any topic category should trigger a content review for that category within 10 business days.
▶ Data leakage Cyber & Data Security
Mitigation Strategy
Customer PII is stripped from messages before transmission to the ChatCore AI API using a pre-processing anonymisation step. Account numbers, NRIC, and contact details are detected via regex and replaced with placeholders. The anonymisation layer is tested monthly against a set of synthetic PII-containing messages.
Appropriate Use Guidance
The anonymisation ruleset must be reviewed quarterly and updated to reflect any new PII formats identified in production logs.

Cybersecurity and Data Protection

Data Shared with Provider
Only the anonymised user message, the retrieved knowledge base chunks, and the FI-defined system prompt are transmitted to the ChatCore AI API. No customer account data, PII, or authentication tokens are included. Data is transmitted over TLS 1.3.
Data Handling by Provider
ChatCore AI does not retain conversation data beyond the duration of the API request. No conversation content is used for model training. ChatCore AI's data processing is governed by a Data Processing Agreement (DPA) executed with the FI. Data residency: Singapore region only.
Cybersecurity Metrics and Results
Penetration test completed March 2026 (external vendor). No critical findings. Two medium findings (session fixation, verbose error messages) remediated in v1.4.0. Jailbreak resilience tested using a library of 200 known jailbreak prompts - 98.5% blocked by input filter. Remaining 1.5% produced out-of-scope but non-harmful responses.
Security and Data Protection Attestations
ChatCore AI holds SOC 2 Type II (last audit: December 2025). ISO/IEC 27001 certified (cert. #: CC-2025-0871). FI's own deployment infrastructure is covered under the FI's existing ISO/IEC 27001 certification.

Data Flow Description
1. Customer sends a message via the web or mobile UI.
2. The PII anonymisation layer strips any detected personal data from the message.
3. The input filter assesses the anonymised message for jailbreak attempts and off-topic content. Blocked messages return a standard deflection response.
4. The retrieval layer queries the vector database with the user message and returns the top 5 most relevant knowledge base chunks.
5. The anonymised message, retrieved chunks, and FI system prompt are sent to the ChatCore LLM API.
6. The generated response is assessed by the output filter for toxicity and policy compliance. Responses failing the filter trigger a fallback response and human handoff.
7. The approved response is returned to the customer UI.
8. The full interaction (pre-anonymisation message, filter verdicts, response, BERTScore) is written to the FI's internal audit log.
